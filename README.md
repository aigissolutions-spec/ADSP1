# ADSP1

**Safety-first AI dev studio and OpenClaw command center**

ADSP1 is a local-first, multi-provider AI workspace designed to make agent use safer, more affordable, and easier to manage.

It is built as a practical control center for OpenClaw-class agent workflows, with a focus on clear runtime truth, bounded execution, provider flexibility, and launch-ready usability.

## What ADSP1 is

ADSP1 is:

- a control center for OpenClaw-style agent use
- a multi-provider workspace
- a local-first and cloud-capable runtime environment
- a safer operator layer
- a setup and continuity layer
- a future governed and trackable environment for sensitive systems

ADSP1 is not:

- a generic chatbot
- a copy of OpenClaw
- a pure dashboard
- an enterprise workflow engine

## Core product direction

The product is being built around five main strengths:

- **Safety-first runtime control**
- **Affordable token strategy through local-first usage**
- **Ease of setup and ease of use**
- **Continuity across provider and model changes**
- **A future governed and trackable path for sensitive systems**

## Product pillars

### 1. Safety-first runtime environment

ADSP1 is being built so that agent power is available without making unrestricted access the default.

Key safety goals include:

- clear **Sandboxed** vs **Unrestricted** runtime modes
- filesystem access restrictions
- allowlist-based folder and file access
- visibility into what the runtime is allowed to touch
- clear warnings when leaving safe mode
- a future governed mode foundation for sensitive systems

### 2. Efficient token strategy / affordability

The guiding rule is:

> Everything should start cheap/local unless the user explicitly escalates.

Key goals include:

- local-first cheap defaults
- premium escalation only when needed
- visible provider/model truth
- no hidden remote fallback for cheap modes
- reduced token burn through summaries and smarter context handling

### 3. Setup wizard / guided onboarding

Ease of setup is intended to be one of ADSP1’s headline strengths.

The setup flow is being designed to detect, guide, and validate:

- OpenClaw
- OpenAI CLI
- Claude CLI
- local LLM runtimes such as Ollama
- default provider/model setup
- recommended first runtime mode

### 4. Core product surfaces

The main product surfaces include:

- **Direct Session** — stable chat/control surface
- **Playground** — runtime testing and experimentation surface
- **Provider Management** — setup, enable/disable, and connection truth
- **Basic run visibility** — status, provider/model used, and errors
- **Basic session history** — recent sessions and transcript reopening

### 5. Continuity and memory

Continuity is important, but it follows safety and affordability in the roadmap.

The long-term direction is to keep identity and thread continuity durable even when providers or models change.

## Design rules

ADSP1 follows these rules:

- safety comes before affordability
- affordability comes before advanced continuity
- continuity should be provider-agnostic and model-agnostic
- setup and ease of use are major product strengths
- power should be available, but unrestricted access should always be deliberate, visible, and easy to bound
- the community version should not be built in a way that blocks a future governed/trackable version
- default to cheap/local where safe and practical
- always show truth clearly: provider, model, mode, restrictions, and runtime posture
